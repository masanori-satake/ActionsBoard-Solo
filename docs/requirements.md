# 要件定義・基本設計仕様書：コンテキスト指向型 GitHub Actions 監視 Chrome 拡張機能

本ドキュメントは、Bamboo（Data Center版）から GitHub Actions への移行、およびその後の恒久的な運用フェーズにおいて、開発チームの生産性を最大化し「無駄な待ち時間」をゼロにするための Chrome 拡張機能の要件と画面構成を定義したものです。

---

## 1. プロジェクト背景と目的

### 1.1 移行に伴うパラダイムシフトと課題

オンプレミスな Bamboo（Data Center版）からクラウド（従量課金ハイブリッド含む）の GitHub Actions への移行において、システム構造の根本的な違い（アーキテクチャ・ギャップ）がチームの認知負荷となります。

* **Bamboo:** 独立した「ビルドプラン」が上層にあり、ソースコードはそこから参照される構造。
* **GitHub Actions:** 「リポジトリ（ソースコード）」が主であり、ワークフロー（YAML）は各リポジトリに内包・結合される構造。

この「リポジトリ結合型」への変化により、複数リポジトリを跨ぐ大規模プロジェクトでは情報が分散し、公式UIだけでは全体像の把握が困難になります。

### 1.2 コアミッション

1. **開発を止めない（クリティカルバグの即時検知）:** 共有ブランチの崩壊を秒単位で検知し、被害の拡大を防ぐ。
2. **ボーッと待つ時間をゼロにする（手空き時間の排除）:** テストや Pages デプロイの完了を能動的にプッシュ通知し、開発者が「待ち時間」に見逃しなく次のタスクへ移れるようにする。

## 2. システム設計思想：アプローチの対比

本ツールが採用する「コンテキスト指向型」**と、一般的な**「無機的一覧型」の対比から、本設計の優位性を明確にします。

| 比較軸 | ❌ A案：無機的一覧型ダッシュボード（不採用） | ⭕ B案：コンテキスト指向型（本設計で採用） |
| --- | --- | --- |
| **画面の思想** | リポジトリとYAMLファイルをそのままフラットに並べる。 | ユーザーが「担当領域」ごとの仮想ワークスペースを作る。 |
| **情報のノイズ** | 80プランすべてが等価に並び、自分に関係のない通知も埋もれる。 | 自分の担当、およびお気に入りだけにノイズレス化される。 |
| **Bamboo脳への配慮** | 構造が違いすぎて、Bamboo慣れしたメンバーが迷子になる。 | 「目的別のプラン管理」に近い操作感を仮想的に再現。 |
| **ライフサイクル** | Actions の `status`（成功/失敗）の判定のみで終了。 | Actions ➔ Pages デプロイ完了まで一気通貫で追跡。 |

### なぜ「Chrome拡張機能（ローカル完結型ポーリング）」なのか

社内ネットワーク制限（プロキシ、ファイアウォール）が厳しいエンタープライズ環境、および将来的なストア一般公開を見据え、Webサーバーや BaaS（Firebase等）を一切挟まない「Vanilla JS + 標準GitHub APIによるクライアントサイド・ポーリング」を採用します。これにより、セキュリティ審査を容易にし、インフラ維持コストを完全ゼロにします。

## 4. GitHub Pages 連携機能（デプロイ追跡）

GitHub Pages 特有の「2段階デプロイ（Actions 成功 ➔ Pages 内部ビルド）」によるタイムロスを解消するための、API 連携要件です。

### 4.1 シーケンス

1. 拡張機能が `GET /repos/{owner}/{repo}/actions/runs` をポーリングし、対象ワークフローの `conclusion: success` を検知。
2. 自動的にターゲットを Pages API（`GET /repos/{owner}/{repo}/pages/deployments`）に切り替え、最新のデプロイステータスを追跡。
3. `status: deliverable`（反映完了）を検知した瞬間、画面のインジケーターを 🟡 から 🟢 に変更し、デスクトップ通知を発火。

## 5. 主要機能要件（UX・見逃しゼロの仕組み）

### 5.1 賢いタイマー制御とレートリミット対策

* **`chrome.alarms` の利用:** バックグラウンド（`service_worker`）での実行を制御。
* **アクティブ・アダプティブ・ポーリング:** * ダッシュボード画面（タブ）が開いている時：30秒間隔で詳細に更新。
* 閉じている時（バックグラウンド）：3分〜5分間隔に落とし、`main` ブランチの破壊と自分のジョブ完了のみをミニマムに監視。APIの制限（1時間5,000回）を余裕で回避。



### 5.2 パーソナライズ機能

* **お気に入り・エイリアス機能:** 全80プランから必要なものを「☆」で取捨選択。味気ない `dashboard-dev.yml` に対し、ツール上で「【本番】E2E自動テスト」のような人間が理解しやすい別名（エイリアス）を付与可能（`chrome.storage.local` に保存）。
* **アイコンバッジの連動:** ブラウザ右上にある拡張機能アイコンのバッジに、現在「失敗したまま放置されているワークフロー」の数を常時赤数字で表示。

## 6. 技術スタックと初期開発ロードマップ

* **技術スタック:** Vanilla JavaScript (ES6+), Material Design 3 (CSS/Components), Chrome Extension API (Manifest v3)
* **次フェーズ（実装の第一歩）:**
指定された複数リポジトリ（ワークスペース定義）から、Actions の最新情報と Pages のデプロイステータスを並行して `fetch` し、1つの統合オブジェクトとしてパースする「コア・データアクセスロジック」の構築。
