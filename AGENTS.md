# ActionsBoard-Solo: AIエージェントおよび開発者ガイド (AGENTS.md)

このドキュメントは、ActionsBoard-Solo の設計思想、開発ルール、および AI エージェントや開発者が開発・保守を行う際の具体的な行動指針をまとめたものです。

## 1. はじめに

本プロジェクトは Public なオープンソース・プロジェクトとして公開されており、不特定多数のユーザーが参照することを前提としています。
ユーザーの認知負荷を最小化し、データの透明性と安全性を最大限に高めることを最優先としています。

### ActionsBoard-Solo の位置づけ

ActionsBoard-Solo は、GitHub Actions の実行状態を「自分（開発者）」のコンテキストに合わせて最適化して表示する、生産性向上ツールです。

- **狙い:** 複数リポジトリの Actions と Pages の状態を一つのダッシュボードで把握し、待ち時間をゼロにすること。
- **非目標:** 全リポジトリの無差別な監視、複雑な CI/CD 管理。

## 2. 設計思想の要約

詳細な要件と設計については、[docs/requirements.md](docs/requirements.md) を参照してください。

### コア・フィロソフィーの要点

- **持続可能性 (Sustainability):** 外部サーバーに依存せず、ブラウザ内（オフライン）で完結すること。
- **ミニマリズム (Minimalism):** 必要十分な機能に絞り、情報のノイズを最小化すること。
- **透明性 (Transparency):** データの保存先や通信先を明示し、ユーザーが安心して利用できること。
- **保守性・長寿命 (Maintainability & Longevity):** 外部フレームワークを一切使用せず、標準的な Web API (Vanilla JS) のみで構築すること。

## 3. 開発・行動ルール

AI エージェントおよび開発者は、以下のルールを厳格に遵守しなければなりません。

### 3.1. 深い計画プロトコル (Deep Planning Protocol) - 対話の原則

実行前に「絶対的な確信」を得るための対話を行う。

- 1. すべての前提条件を確認するために質問を投げ、要件を完全に把握する。
- 2. ユーザーの意図や「何を実現したいか（Desire）」に焦点を当てて質問する。
- 3. 不明点がゼロになるまで深掘りした質問を継続する。
- 4. ユーザー情報の要約やレポート生成においては、中立的な立場を保ち、三人称（「ユーザー」または「Masanori SATAKE」）を使用せよ。一人称（「私」）や二人称（「あなた」）の使用は避けること。
- 5. すべてのコミュニケーションは `request_user_input` または `message_user` を通じて行い、独断を避ける。
- 6. 確信を得た後、`set_plan` で計画を立て、承認を得てから自律的な実行へ移る。

### 3.2. 言語統制 (Language Control) - 最優先事項

あらゆる外部コミュニケーションにおいて**日本語を最優先で使用し、英語の使用を最小限に留めなければならない**。

以下のすべての場面において、日本語を使用せよ：

- **対話:** `request_user_input` や `message_user` によるチャット。
- **PR作成:** タイトル（`title`）および説明文（`description`）。
- **コミットメッセージ:** Git のコミットメッセージ（`feat: ` などのキーワード以降）。
- **コメント返信:** レビューに対する回答。

### 3.3. 公開プロジェクトとしての指針

- **無保証 (No Warranty):** 本ソフトウェアは個人による提供であり、無保証であることを README.md や About 画面で一貫して明示し続けなければならない。

### 3.4. 実装・保守の原則

- **仕様への忠実性:** すべての変更は `requirements.md` の設計思想、セキュリティ標準、および品質基準に合致しなければならない。
- **Vanilla JS の徹底:** 外部ライブラリやフレームワーク（React, Vue 等）は使用せず、標準 Web API のみを使用する。
- **CSS:** 提供された `material3.css` をベースに、一貫した Material Design 3 スタイルを適用する。
- **ディレクトリ構造:** `projects/app` 配下にソースコードを配置する構造を厳格に守る。

## 4. 設計原則 (Design Principles)

- **KISS (Keep It Simple, Stupid):** シンプルさを保つ。
- **DRY (Don't Repeat Yourself):** 重複を避ける。
- **SLAP (Single Level of Abstraction Principle):** 抽象化レベルを揃える。
- **YAGNI (You Ain't Gonna Need It):** 今必要ないものは作らない。
